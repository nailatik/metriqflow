## Слайд 7 — Архитектура (клиент ↔ сервер)

**Визуал:** компонентная диаграмма (из работы, fig:architecture). Анимировать поток данных стрелками.

```
  ┌──────────┐  REST/JSON  ┌──────────┐   SQL   ┌────────────┐
  │ Frontend │ ──────────► │ Backend  │ ──────► │            │
  │ Next.js  │ ◄────────── │ Express  │ ◄────── │ PostgreSQL │
  └──────────┘             └──────────┘         │            │
                                                 │ (общая БД) │
  ┌──────────┐  MTProto    ┌──────────┐   SQL   │            │
  │ Telegram │ ◄────────── │ TG-бот   │ ──────► │            │
  │ MTProto  │             │ Python   │         └────────────┘
  └──────────┘             └──────────┘
```

Тезисы:
- 3 независимых процесса, общая точка — БД
- Браузер **не имеет** прямого доступа к БД (только REST)
- Бот на Python — **техническая необходимость**: Telethon/MTProto есть только для Python

> Объяснить, почему бот отдельно — это сильный честный аргумент, не «костыль».

## Слайд 8 — Сбор данных и метрики

**Визуал:** слева формула ER крупно, справа — тепловая карта активности (`photo/8.png`).

$$ \mathrm{ER} = \frac{\text{реакции} + \text{репосты} + \text{комментарии}}{\text{просмотры}} \times 100\% $$

Тезисы:
- TG: Bot API (общение) + Telethon/MTProto (статистика постов)
- VK: `wall.get` через сервисный токен
- Фоновый планировщик собирает данные по расписанию, уважая rate-limits
- 4 периода анализа: 24ч / 7д / 30д / всё время + тепловая карта «лучших часов»

## Слайд 10 — AI-инсайты: почему им можно верить (главный технический козырь)

**Визуал:** два скрина рядом — `photo/12.png` (много данных, уверенный совет) vs `photo/13.png` (мало данных, AI честно сдержан).

**Суть: бэкенд не даёт модели врать.**

- **Data-tier** (`low`/`medium`/`high`) считается детерминированно на бэкенде по числу постов — *до* запроса к модели
- `gateBestTime`: совет о лучшем времени публикации **блокируется**, если данных мало (нельзя советовать «понедельник 18:00» по одному посту)
- `data_quality` вычисляется на сервере и передаётся как контекст — **модель не может его переопределить**

> Это слайд, который отделяет «прикрутил ChatGPT» от «инженерно выстроил доверие к LLM». Произнести медленно.

## Слайд 11 — Авто-отчёты

**Визуал:** скрин модалки авторепорта (`photo/11.png`) + иконки форматов CSV / PDF / XML + иконка Telegram-доставки.

Тезисы:
- Ручная генерация и доставка по расписанию — **одна** функция `generateFile`
- Форматы: CSV, XML (вручную), PDF (PDFKit, кириллица)
- `node-cron` каждую минуту, `next_send_at` считается в SQL с учётом часового пояса
- Доставка в Telegram-бот или на e-mail
- **Никто из конкурентов не доставляет отчёт прямо в Telegram**

## Слайд 12 — Безопасность (production-grade)

**Визуал:** щит в центре, вокруг — 5 «замков» с подписями.

- JWT в **httpOnly-куки** → недоступен из JS (защита от XSS)
- Refresh-токены с ротацией, в БД только **bcrypt-хеш**
- **Helmet** + CSP, **rate limiter** на middleware
- **Zod** валидирует всё на входе, фронт — не последний арбитр
- Согласие на обработку ПДн (152-ФЗ) на уровне схемы

**Якорь:** *Безопасность — не слой поверх, а часть архитектуры с первого дня.*

## Шпаргалка по защите от вопросов (бэкап-слайды / в уме)

- **«Почему без ORM?»** → прямой SQL = полный контроль над запросами, видно что уходит в БД, нет скрытой абстракции.
- **«Почему бот отдельным процессом?»** → Telethon (MTProto) только для Python, на Node.js аналога нет. Это не архитектурный каприз, а ограничение экосистемы.
- **«VK stats.get?»** → закрыт для внешних приложений, используем только публичные метрики `wall.get`. Ограничение зафиксировано в коде честно.
- **«Не галлюцинирует ли AI?»** → data-tier + gateBestTime + серверный data_quality. Модель физически не получает данных, по которым нельзя сделать вывод.
- **«Сколько стоит AI?»** → Haiku + кэш 6ч + квоты Metriqs по тарифам. Просмотр кэша бесплатен.
- **«Чем лучше LiveDune?»** → TG+VK вместе + AI + доставка в Telegram + дешевле (590₽).
