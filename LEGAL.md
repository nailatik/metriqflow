# LEGAL.md — Юридические документы и согласия (152-ФЗ)

Спецификация для исполнителя. Цель: привести регистрацию и публичные страницы Metriq Flow
в соответствие с 152-ФЗ «О персональных данных» (ред. с изменениями, действующими с 01.09.2025),
38-ФЗ «О рекламе» и 242-ФЗ (локализация). Сделать публичные правовые страницы,
раздельные согласия на регистрации и аудит-лог согласий.

> Дисклеймер: ниже — корректная по структуре и compliance-маппингу схема + шаблоны.
> Финальные тексты документов перед публикацией стоит вычитать юристом и подставить реквизиты ИП.
> Архитектура (раздельные согласия, лог, версионирование) — обязательна и не зависит от вычитки.

---

## 0. TL;DR

- **3 документа**: Политика обработки ПДн, Согласие на обработку ПДн, Пользовательское соглашение (оферта).
- **2 обязательных чекбокса** на регистрации (раздельные, не pre-checked) + **1 опциональный** (рассылка).
- **Нельзя** объединять согласие на ПДн с офертой в одну галочку (поправки с 01.09.2025).
- Добавить **аудит-лог согласий** (кто, когда, IP, версия документа) — миграция `021_consents.sql`.
- Текущее состояние: 1 чекбокс `agreedToProcessing`, один boolean `users.agreed_to_processing`, лога нет.

### Маппинг документ → галочка

| # | Документ | Маршрут | Обяз. | Чекбокс при регистрации |
|---|----------|---------|-------|--------------------------|
| 1 | Политика обработки ПДн | `/legal/privacy` | да (ст.18.1 ч.2) | **нет** — только публикация + ссылка в футере |
| 2 | Согласие на обработку ПДн | `/legal/consent` | да | **да**, обязательный, blocking |
| 3 | Пользовательское соглашение (оферта) | `/legal/terms` | да | **да**, обязательный, blocking |
| 4 | Согласие на рекламную рассылку | секция в `/legal/consent` | если есть промо-письма | опциональный, не блокирует |

## 1. Правовая основа (зачем каждый кусок)

- **152-ФЗ ст.18.1 ч.2** — оператор обязан опубликовать документ, определяющий политику обработки ПДн,
  в открытом доступе (для сайта = публичная страница). → Политика.
- **152-ФЗ ст.9 + поправки, действующие с 01.09.2025** — согласие должно быть конкретным, предметным,
  информированным, сознательным, однозначным и **оформляться отдельно** от иных соглашений (нельзя «зашивать»
  в оферту/кнопку без отдельного чекбокса). → отдельная галочка на Согласие, отдельная на Оферту.
- **38-ФЗ ст.18** — реклама по сетям связи (e-mail/SMS-рассылки) только с предварительного согласия,
  оформленного **отдельно** от прочих согласий. → отдельный опциональный чекбокс.
- **242-ФЗ** — ПДн граждан РФ хранить в БД на территории РФ. → инфраструктурный флаг (см. §7).
- **420-ФЗ (штрафы, с 30.05.2025)** — оборотные/повышенные штрафы за обработку без согласия,
  отсутствие политики, неуведомление РКН. → мотивация.
- **Уведомление РКН** — оператор обязан уведомить Роскомнадзор до начала обработки. → действие пользователя (§7), не код.

## 2. Frontend — публичные правовые страницы

### 2.1 Маршруты

Новая route-группа `app/[locale]/(legal)/`:

```
app/[locale]/(legal)/
  layout.tsx                      // общий контейнер: max-w-3xl, типографика, шапка/футер лендинга
  privacy/page.tsx                // Политика обработки ПДн
  consent/page.tsx                // Согласие на обработку ПДн (+ секция о рассылке)
  terms/page.tsx                  // Пользовательское соглашение / оферта
```

- **Server components** (как `register/page.tsx`). Контент — статический JSX или markdown-рендер.
- `generateMetadata`: `robots: { index: true, follow: true }` — в ОТЛИЧИЕ от auth-страниц (там noindex).
  Правовые страницы должны индексироваться и быть публично доступны.
- Используют `getTranslator(locale, ...)` и `Link` из `@/i18n/navigation` для консистентности.
- Layout переиспользует шапку/футер лендинга (см. `widgets/Header`, `features/landing/ui/Footer`),
  чтобы страницы выглядели частью сайта, а не голым текстом.

### 2.2 Язык документов

Юридически обязывающая версия — **на русском** (оператор = ИП в РФ, применяется 152-ФЗ).
Рекомендация: контент документов рендерить **на русском независимо от UI-locale**
(в начале EN-версии — плашка «Informational translation. The Russian version prevails»),
либо на первом этапе RU-only с доступом из обеих локалей. Не делать машинный перевод юр. текста как обязывающий.

### 2.3 Версионирование документов

Каждый документ имеет версию и дату вступления — нужно, чтобы в логе согласий хранить, **с какой именно
редакцией** согласился пользователь. Завести единый источник:

```ts
// frontend/shared/lib/legal.ts  (и зеркало в backend/src/config/legal.ts)
export const LEGAL_VERSIONS = {
  privacy: { version: "1.0", date: "2026-06-16" },
  consent: { version: "1.0", date: "2026-06-16" },
  terms:   { version: "1.0", date: "2026-06-16" },
} as const;
```

Версии показывать в подвале каждого документа: «Редакция 1.0 от 16.06.2026».

## 3. Frontend — регистрация (RegisterForm)

Файл: `frontend/features/auth/ui/RegisterForm/RegisterForm.tsx`, step 2.

### 3.1 Заменить один чекбокс на блок из 2–3

Текущее (удалить — единственный `agreedToProcessing`):

```tsx
formData: { ..., agreedToProcessing: false }
// один <input checkbox id="agreed"> + errors.agreement
```

Новое — `formData`:

```ts
const [formData, setFormData] = useState({
  fullName: "", birthDate: "", organization: "", phone: "",
  agreedToProcessing: false,  // ПДн — обязательно
  agreedToTerms: false,       // оферта — обязательно
  agreedToMarketing: false,   // рассылка — опционально
});
```

Новый UI-блок (заменяет текущий единственный чекбокс):

```tsx
{/* 1. ПДн — обязательно */}
<Checkbox checked={formData.agreedToProcessing} onChange={...} error={errors.agreement}>
  {t.rich("step2.consentPdn", {
    consent: (c) => <Link href="/legal/consent" target="_blank">{c}</Link>,
    privacy: (c) => <Link href="/legal/privacy" target="_blank">{c}</Link>,
  })} *
</Checkbox>

{/* 2. Оферта — обязательно (ОТДЕЛЬНАЯ галочка, не объединять с п.1) */}
<Checkbox checked={formData.agreedToTerms} onChange={...} error={errors.terms}>
  {t.rich("step2.consentTerms", {
    terms: (c) => <Link href="/legal/terms" target="_blank">{c}</Link>,
  })} *
</Checkbox>

{/* 3. Рассылка — опционально, не блокирует submit */}
<Checkbox checked={formData.agreedToMarketing} onChange={...}>
  {t("step2.consentMarketing")}
</Checkbox>
```

> Все чекбоксы по умолчанию `false` (не pre-checked) — требование закона.
> Создать общий `shared/ui/Checkbox` (сейчас в форме сырой `<input type=checkbox>`), чтобы переиспользовать.

### 3.2 Валидация (handleStep2)


## 6. Backend — приём и хранение согласий

### 6.1 Schema/тип (`src/schemas/auth.schemas.ts`, `src/controllers/auth.controller.ts`)

```ts
// zod
agreedToProcessing: z.literal(true, "Consent to data processing is required"),
agreedToTerms:      z.literal(true, "Acceptance of terms is required"),
agreedToMarketing:  z.boolean().optional().default(false),
```

`AuthBody` дополнить полями `agreedToTerms`, `agreedToMarketing`.
Добавить проверку `if (!agreedToTerms) return 400`.

### 6.2 Миграция `migrations/021_consents.sql`

Цель — хранить **доказательство** согласия (152-ФЗ требует подтверждать факт и условия согласия).
Сейчас `users.agreed_to_processing boolean` недостаточно: нет момента, IP, версии документа, согласия на оферту/рассылку.

```sql
-- флаги на users (быстрый доступ к текущему состоянию)
ALTER TABLE users ADD COLUMN IF NOT EXISTS agreed_to_terms     boolean NOT NULL DEFAULT false;
ALTER TABLE users ADD COLUMN IF NOT EXISTS agreed_to_marketing boolean NOT NULL DEFAULT false;

-- аудит-лог: по строке на каждое согласие/отзыв
CREATE TABLE IF NOT EXISTS user_consents (
  id           bigserial PRIMARY KEY,
  user_id      integer NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  consent_type text    NOT NULL,           -- 'pdn' | 'terms' | 'marketing'
  doc_version  text    NOT NULL,           -- из LEGAL_VERSIONS, напр. '1.0'
  granted      boolean NOT NULL,           -- true = дал, false = отозвал
  ip           inet,
  user_agent   text,
  created_at   timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_user_consents_user ON user_consents(user_id, consent_type, created_at DESC);
```

Применить через psql (homebrew, vars из `backend/.env`) — см. рабочий процесс миграций проекта.

### 6.3 Контроллер register — писать лог

В `register` (`src/controllers/auth.controller.ts`, после успешного INSERT users, в той же транзакции по возможности):

```ts
// users INSERT: добавить agreed_to_terms, agreed_to_marketing в колонки/values
// затем:
const ip = req.headers["x-forwarded-for"]?.toString().split(",")[0]?.trim() || req.socket.remoteAddress;
const ua = req.headers["user-agent"] || null;
const rows = [
  [user.id, "pdn",   LEGAL_VERSIONS.consent.version, true, ip, ua],
  [user.id, "terms", LEGAL_VERSIONS.terms.version,   true, ip, ua],
];
if (agreedToMarketing) rows.push([user.id, "marketing", LEGAL_VERSIONS.consent.version, true, ip, ua]);
// bulk insert в user_consents
```

Завести `backend/src/config/legal.ts` с теми же версиями, что на фронте (единый источник правды по версиям).

### 6.5 Отзыв согласия (минимально)

## 8. Шаблоны документов (контент)

Плейсхолдеры: `{{OPERATOR_NAME}}`, `{{INN}}`, `{{OGRNIP}}`, `{{ADDRESS}}`, `{{EMAIL}}`, `{{SITE_URL}}`, `{{EFFECTIVE_DATE}}`.

### 8.1 Политика обработки персональных данных (`/legal/privacy`)

Разделы:
1. Общие положения — оператор ({{OPERATOR_NAME}}, ИНН {{INN}}), правовые основания (152-ФЗ), что это публичный документ по ст.18.1 ч.2.
2. Термины — ПДн, обработка, субъект, оператор и т.д.
3. Какие ПДн обрабатываются — e-mail, ФИО, дата рождения, телефон, организация; токены/данные подключённых соцсетей (VK и др.); платёжные данные (через ЮКассу, оператор карт не хранит).
4. Цели обработки — регистрация и доступ к сервису, оказание услуг аналитики, биллинг, уведомления, поддержка.
5. Правовые основания и условия — согласие субъекта, исполнение договора (оферты).
6. Порядок и сроки обработки, хранение в РФ (242-ФЗ).
7. Передача третьим лицам / трансграничная — обработчики (платёжный провайдер, e-mail-релей), без трансграничной передачи (или с указанием).
8. Права субъекта — доступ, уточнение, удаление, отзыв согласия; как реализовать (написать на {{EMAIL}}).
9. Меры защиты — шифрование токенов, хэш паролей, ограничение доступа.
10. Cookie и аналитика.
11. Изменения политики, контакты, дата редакции.

### 8.2 Согласие на обработку персональных данных (`/legal/consent`)

- Субъект, давая согласие при регистрации, соглашается с тем, что {{OPERATOR_NAME}} обрабатывает перечисленные ПДн
  в указанных целях, перечисленными способами, на срок действия аккаунта + N лет / до отзыва.
- Перечень ПДн и действий (сбор, запись, хранение, использование, удаление…).
- Способ отзыва — письмо на {{EMAIL}} либо удаление аккаунта.
- **Отдельная секция (38-ФЗ): согласие на рекламную рассылку** — даётся опциональным чекбоксом, может быть отозвано
  в любой момент (ссылка отписки в письмах + тумблер в профиле). Явно: отказ от рассылки не влияет на доступ к сервису.

### 8.3 Пользовательское соглашение / Договор-оферта (`/legal/terms`)

Разделы:
1. Термины, стороны (оператор-исполнитель / пользователь-заказчик).
2. Предмет — доступ к SaaS-сервису аналитики соцсетей на условиях оферты.
3. Регистрация и аккаунт — требования, ответственность за доступы/токены соцсетей.
4. Тарифы, оплата, триал — через ЮКассу; порядок списаний/продлений; возвраты.
5. Права и обязанности сторон, допустимое использование.
6. Ограничение ответственности (данные соцсетей предоставляются «как есть», зависимость от API площадок).
7. Интеллектуальная собственность.
8. Изменение условий, расторжение, удаление аккаунта.
9. Применимое право (РФ), порядок споров, реквизиты {{OPERATOR_NAME}}.
